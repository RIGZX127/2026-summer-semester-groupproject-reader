一、Phase 1 做了什么（用直白的语言）

Phase 1 的目标是搭建整个项目的"地基"——在还没有任何界面的情况下，把数据怎么存、RSS 怎么拉取、文章怎么写进数据库这三件核心的事情做完，并用测试证明它们能正确运行。

▸ 做了三件具体的事

M1.1 项目骨架
依赖配置 + 目录结构 + 全局状态单例
M1.2 数据库层
10 张表 + 版本化迁移 + Feed/Entry CRUD 接口
M1.3 Feed 解析与同步
RSS 解析 + 并发同步 + OPML 导入导出

▸ 对后续跨平台迁移的作用

整个数据层（store/）和核心逻辑层（core/）使用的全是纯 Python 标准库或跨平台库，不依赖任何操作系统特有的 API。这意味着：

  • 数据库文件格式（SQLite）在 Windows、macOS、Linux 上完全一致，同一个 .db 文件可以跨系统打开
  • RSS 解析（feedparser）、HTTP 请求（httpx）、OPML 处理均为跨平台库，不需要任何改动
  • 成员 C 在 Phase 1.4 之后做的 UI（PySide6/Qt）也是跨平台的，相当于界面层"插"在这个地基上
  • 未来如果要换 UI 框架，数据层和业务逻辑层完全不需要动

▸ 可能会遇到的矛盾

矛盾点具体描述风险等级
路径分隔符 Windows 用反斜杠 \，macOS/Linux 用斜杠 /。app/app.py 里的默认数据库路径已用 pathlib.Path 处理，但后续代码需要保持这个习惯 中
Windows Long Path 限制 本次开发中 PySide6-Essentials 因路径超 260 字符无法安装，导致 PySide6.QtCore 无法正常加载。测试时用降级方案绕过，但 UI 运行时需要完整安装 中
SQLite 并发写入 内存数据库（:memory:）在多个异步任务并发写入时存在锁竞争。生产环境用磁盘 DB + WAL 模式 + timeout=30 可承受，但极高并发场景需关注 中
文件系统大小写 Windows 文件系统不区分大小写，macOS/Linux 区分。模块导入路径如果大小写写错，在 Windows 上正常运行，迁移到 Linux CI 时会报 ModuleNotFoundError 中
platform 包名冲突 项目的 platform/ 目录遮蔽了 Python 标准库的 platform 模块，已在 __init__.py 中修复，但需注意后续代码不要直接 import platform 低

五、开发中遇到并解决的问题

问题 1：PySide6.QtCore 无法安装（Windows Long Path 限制）
PySide6-Essentials 的安装路径超过 Windows 260 字符限制，导致 QtCore 子模块缺失，同步模块无法 import。
解决：将 PySide6 的 import 改为 try/except 懒加载，测试环境自动降级为轻量回调类，生产环境有完整 PySide6 时自动使用 Qt Signal。
问题 2：同步失败计数错误（sync_all 计不到 total_failed）
sync_feed 内部捕获了 FeedParseError 并返回 0，sync_all 的 gather 收不到异常，total_failed 永远为 0。
解决：引入哨兵值 _SYNC_FAILED = -1，失败时返回 -1，sync_all 检查 result == -1 计入失败数，区分"成功但无新文章(=0)"与"失败(-1)"。
问题 3：内存数据库不支持 WAL 模式
SQLite 规范规定 :memory: 数据库的 journal_mode 固定返回 "memory"，测试断言 WAL 会失败。
解决：拆分为两个测试用例：磁盘临时文件验证真正的 WAL，内存数据库只验证其他功能。
问题 4：platform/ 包遮蔽标准库 platform 模块
项目目录下的 platform/__init__.py 导致 import platform 时找到项目包而非标准库。
解决：在 platform/__init__.py 中用 importlib 从标准库路径直接加载真正的 platform 模块并重新导出。
